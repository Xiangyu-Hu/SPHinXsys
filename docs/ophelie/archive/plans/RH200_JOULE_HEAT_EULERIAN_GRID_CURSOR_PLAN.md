# RH200 OPHELIE JouleHeat Eulerian Grid Coupling Plan

## 0. Current decision

The current `em-fixed` coupling is physically wrong for the RH200 stirring demonstration.

At present, after the OPHELIE electromagnetic solve, `JouleHeat` is written directly as a particle variable. During stirring, particles carry their original `JouleHeat` values with their Lagrangian labels. This means the heat source moves with the material particles, which is not the correct one-way electromagnetic heating interpretation when the coil is fixed and the electrical conductivity is fixed.

The correct midterm-demonstration model should be:

\[
Q = Q(\mathbf{x})
\]

where `Q` is a fixed laboratory-frame spatial field generated by the fixed induction coil. The moving SPH particles should sample this spatial field at their current positions:

\[
\rho c_p \frac{D T_i}{D t}
= \nabla \cdot (k \nabla T)_i + Q(\mathbf{x}_i(t)).
\]

Therefore, for the RH200 intermediate delivery, the formal path is:

```text
OPHELIE EM solve once on the initial glass particle distribution
    ↓
deposit particle JouleHeat onto a fixed Eulerian grid covering full_oil.stl
    ↓
during stirring/thermal simulation, each particle samples Q_grid(x_i(t))
    ↓
T_i += Q_grid(x_i(t)) * dt / (rho_i * cp)
```

This replaces the current particle-carried `em-fixed` heat source with a spatially fixed Eulerian heat-source field.

---

## 1. Why the current particle-attached JouleHeat is wrong

The current implementation effectively does:

\[
Q_i(t) = Q_i(0) = Q(\mathbf{x}_i^0).
\]

After stirring starts, particles still carry the Joule heat value assigned at their initial position. This creates a material-label heat source. The observed blocky/checkerboard JouleHeat pattern in the stirred RH200 view is consistent with this issue: the color patches follow particles instead of remaining fixed in space.

For a fixed coil and fixed conductivity, the electromagnetic source should remain fixed in the laboratory coordinate system. Particle motion should only advect temperature, not the electromagnetic heat-source pattern.

The corrected relation should be:

\[
Q_i(t) = Q(\mathbf{x}_i(t)).
\]

---

## 2. Use `full_oil.stl` as the grid domain

The Eulerian JouleHeat grid should be constructed over the complete available stirring domain, not merely over the initial fluid particle bounding box.

Use `full_oil.stl` as the preferred geometry source for the grid bounding box.

Reason:

- `oil.stl` or initial glass particles represent the initial occupied liquid region.
- During stirring, the free surface may rise, deform, or splash locally.
- If the grid only covers the initial liquid particles, particles can move outside the initial EM domain and sample zero or invalid heat.
- `full_oil.stl` represents the full admissible vessel/fluid domain and is a better bounding volume for storing a fixed spatial heating field.

Recommended grid domain rule:

```text
Preferred:
  bbox = bounding box of full_oil.stl

Fallback:
  bbox = initial glass particle bbox
  expand z direction by 10%
  expand x/y by 1–2 grid spacings
```

For the current RH200 case, Cursor should use `full_oil.stl` as the default.

---

## 3. Do not store JouleHeat in the LevelSet narrow band

The `test_2d_static_confinement` case is useful as a reference for how SPHinXsys uses geometry, level-set data, kernel integrals and near-boundary corrections, but it is not the right storage mechanism for this JouleHeat field.

The static confinement/level-set machinery is mainly for:

```text
1. shape geometry;
2. signed distance;
3. surface normal;
4. kernel integral correction;
5. kernel gradient integral correction;
6. near-boundary support completion.
```

It is not designed as a full-domain Eulerian volume-field storage for `Q(x)`.

Reasons not to use LevelSetShape for JouleHeat in this stage:

```text
1. Level-set storage may be narrow-band or sparse-package based.
2. It may not cover the full fluid volume where Q(x) is needed.
3. Extending it to store volume JouleHeat would mix geometry responsibilities with thermal source storage.
4. It would complicate debugging and delay the midterm delivery.
```

Therefore:

```text
Use LevelSet/geometry only for shape, normal, boundary, or static confinement.
Use a separate dense Eulerian grid for JouleHeat Q(x).
```

## 4. New official Joule mode

Keep the current particle-attached mode only for debug comparison, but do not use it for official midterm results.

Recommended modes:

| Mode | Meaning | Status |
|---|---|---|
| `uniform` | Constant JouleHeat on all particles | Debug / energy-closure test |
| `analytic-position` | Recompute analytic Q(x_i) from current position every step | Debug / fake heat field |
| `em-fixed` | Current implementation: EM JouleHeat written once to particles and carried with particles | Debug only; not official |
| `em-grid` | EM JouleHeat deposited to fixed Eulerian grid; particles sample Q(x_i(t)) | Official midterm mode |

Cursor should add:

```bash
--joule-mode=em-grid
```

and make this the recommended mode for the midterm coupled RH200 result.

## 7. Deposition from EM particles to Eulerian grid

After the OPHELIE EM solve and target-power scaling, extract from the glass particles:

```text
Position
JouleHeat
VolumetricMeasure / particle volume
```

Then deposit to the Eulerian grid in host code for the first implementation.

This is acceptable because:

```text
1. EM is solved once or infrequently.
2. Deposition is not done every acoustic step.
3. Host deposition avoids GPU atomic-add complexity.
4. Runtime cost is negligible compared with development risk.
```

Use Cloud-in-Cell / trilinear splatting:

For each particle `i`:

```text
x_i = particle position
Q_i = JouleHeat_i
V_i = particle volume
```

Find the containing grid cell and distribute to the 8 surrounding nodes/cell corners with trilinear weights `w_ig`.

Accumulate:

\[
S_g \mathrel{+}= Q_i V_i w_{ig}
\]

\[
W_g \mathrel{+}= V_i w_{ig}
\]

After deposition:

\[
Q_g = \frac{S_g}{W_g + \epsilon}.
\]

This preserves `Q_g` as power density in `W/m^3`.

Do **not** simply do:

```text
Q_g += Q_i
```

because that would make the grid value depend on local particle number density instead of representing a physical field value.

Empty nodes:

```text
if W_g == 0:
    Q_g = 0
```

Do not implement extrapolation in the first version. If boundary discontinuities are visible later, add optional nearest-valid or one-cell dilation.

## 8. Target-power scaling after grid deposition

The OPHELIE solve may already scale particle JouleHeat to a target total power, e.g. 50 kW:

\[
P_{\mathrm{particle}} = \sum_i Q_i V_i.
\]

However, after deposition and interpolation, the sampled power may shift slightly. Therefore, add a grid-level power check and optional grid rescaling.

Compute:

\[
P_{\mathrm{em,particle}} = \sum_i Q_i V_i
\]

\[
P_{\mathrm{grid,sample}}^0 = \sum_i Q_{\mathrm{grid}}(\mathbf{x}_i^0) V_i
\]

Then, if target power is enabled:

\[
s_{\mathrm{grid}} = \frac{P_{\mathrm{target}}}{P_{\mathrm{grid,sample}}^0}
\]

\[
Q_g \leftarrow s_{\mathrm{grid}} Q_g.
\]

Then recompute:

\[
P_{\mathrm{grid,sample,after}}^0
= \sum_i Q_{\mathrm{grid,scaled}}(\mathbf{x}_i^0) V_i.
\]

Acceptance:

```text
P_grid_sample_initial_after_scale ≈ target_power
```

This makes the heating energy budget consistent with the actual sampled heat source used by the thermal solver.

## 9. Device-side sampling during the thermal/fluid run

During the stirring/thermal simulation, before applying the Joule heat temperature source, sample the fixed grid at the current particle positions.

Add a CK/SYCL-compatible dynamics:

```cpp
class SampleJouleHeatFromGridCK
{
public:
    void update(size_t index_i, Real dt = 0.0)
    {
        Vecd x = pos_[index_i];
        joule_heat_[index_i] = trilinearSample(grid_q_, x);
    }
};
```

Sampling rules:

```text
1. Use 1D contiguous grid array.
2. Index: idx = i + nx * (j + ny * k).
3. Each particle independently samples Q_grid(x_i).
4. If x_i is outside the grid bbox, return Q = 0.
5. Count out-of-grid particles for diagnostics.
```

Then apply the temperature source:

\[
T_i^{n+1} = T_i^n + \Delta t \frac{Q_i}{\rho_i c_p}.
\]

The main loop should do:

```cpp
if (joule_mode == JouleMode::EmGrid)
{
    sample_joule_heat_from_grid.exec();
}

apply_joule_heat_to_temperature.exec(dt);
```

For `analytic-position`, the source is recomputed from current particle position. For `uniform`, no sampling is required. For old `em-fixed`, the particle value is used as-is, but this should be marked debug-only.

## 10. Main-loop placement

Recommended thermal/fluid order for the midterm case:

```text
1. Fluid acoustic step / pressure relaxation.
2. Rotor/wall interaction and rotor motion update.
3. Sample JouleHeat from Eulerian grid at current glass particle positions.
4. Apply Joule heating source to Temperature.
5. Thermal diffusion relaxation.
6. Density/neighbor/configuration update as required by the existing RH200 loop.
7. Output VTP/CSV.
```

The exact order should follow the existing RH200 SPHinXsys loop style, but the key requirement is:

```text
Sample Q_grid(x_i(t)) before applying Joule heat.
Do not use stale particle-carried Q_i from t=0.
```

## 13. Handling free-surface rise / particles outside the grid

Because `full_oil.stl` is used as the grid domain, most physically admissible particle positions should remain within the grid.

For particles outside the grid:

```text
Q = 0
out_of_grid_particle_count += 1
```

This is simple and safe for the first version.

If `out_of_grid_particle_fraction` becomes significant:

```text
1. enlarge the full_oil-based bbox by 1–2 grid spacings;
2. add z margin;
3. inspect whether the fluid motion is physically leaving the vessel;
4. only then consider extrapolation.
```

Do not implement complex extrapolation in the first version.
